Architecture: LLM ↔ SPARQL closed loop

The pipeline runs entirely client-side via WebAssembly (wllama). No data leaves your device.

[ Human Input ]
       │
       ▼
┌──────────────────────┐
│  Llama 3.2 (WASM)    │ ◄── System prompt with QualiaDB ontologies
└──────────┬───────────┘
           │ Emits SPARQL query or JSON-LD
           ▼
┌──────────────────────┐
│  SPARQL Engine       │ ◄── QualiaDB :4242 | Wikidata | Both
└──────────┬───────────┘
           │ Returns semantic bindings
           ▼
┌──────────────────────┐
│  Llama 3.2 (WASM)    │ ──► Natural language answer + JSON-LD node
└──────────────────────┘

If the SPARQL query fails validation, the error is fed back to the model for self-correction (up to 3 retries). All inference runs through the standard wllama Web Worker thread.

COI note: The COI service worker at the top of this page injects Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp headers on reload, enabling SharedArrayBuffer for multi-threaded WASM inference.
