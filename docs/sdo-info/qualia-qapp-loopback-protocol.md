# Qualia Qapp Loopback Protocol

**Status:** Internal Explainer / Draft | **Updated:** 2026-06-12

This document defines the localhost / loopback serving and host-embedded access boundary for Qualia Apps (Qapps). It explains how the sovereign desktop daemon bridges the gap between the secure Rust backend and ephemeral UI clients (Desktop Dioxus panes, Mobile WASM PWAs).

## 1. Introduction

The Qualia-DB daemon runs a local HTTP/WebSocket server (default port `4242`). This server is the singular trust boundary for external UIs, enforcing the Principal's capability grants and providing a standardized API for Qapps.

## 2. Core Endpoints and URL Model

The loopback server exposes three major categories of endpoints:

### 2.1. System and Query
- `GET /health` - Daemon health and telemetry.
- `POST /query` - HTTP SPARQL 1.1 endpoint for querying the local graph.
- `WS /` - General-purpose WebSocket for live UI streaming (e.g. LLM token streams to the mobile PWA).

### 2.2. Cooperative Chat Relay
Enables the SocialWebNet group chat and sub-agent coordination:
- `POST /chat/publish` - Append an encrypted message to the local inbox WAL.
- `GET /chat/pull` - Incremental relay pull using a Lamport watermark.

### 2.3. Torrent and Web Seed (Ontology Distribution)
The daemon acts as a WebTorrent HTTP seed to distribute large `.c.q42` ontology artifacts locally and globally:
- `GET /torrent/webseed/{hash}` - Direct HTTP seed access for artifact chunks.
- `POST /torrent/seed` - Register a new ontology artifact for seeding.
- `POST /torrent/unseed` - Stop seeding an artifact.
- `GET /torrent/telemetry` - Live seeder statistics.
- `POST /torrent/sync` - Reload seeds from the workbench index.

## 3. Trust Assumptions and Lifecycle

### 3.1. Localhost Isolation
By default, the REST and WebTorrent endpoints bind strictly to `localhost` (`127.0.0.1`), utilizing CORS constraints to ensure only the approved Webizen UI (Qualia Studio or validated local clients) can invoke them.

### 3.2. Tethered Mobile Edge (WASM PWA)
When a mobile device connects to the daemon (via SocialWebNet or local network), the protocol enforces:
1. **QR-Scan Bootstrap**: The mobile client scans a QR code generated by the desktop daemon containing the initial cryptographic seed.
2. **Challenge-Response**: The mobile DID authenticates via an Ed25519 signature challenge.
3. **Capability Gating**: The daemon grants a restricted `CapabilityProfile` to the WebSocket connection, preventing unauthorized destructive operations while allowing remote LLM inference and query evaluation.

## 4. Qapp Host Embedding

When a Qapp is launched from the **Qapp Vault**:
1. The daemon unpacks the `.qapp.zip` archive.
2. The daemon verifies the `signature.sig` against the developer's DID.
3. The WASM payload is served from memory via a dynamically generated, sandboxed localhost URL, utilizing Private Network Access headers to isolate the context.
4. The Qapp communicates with the daemon via the established WebSocket protocol, restricted strictly by its embedded `manifest.json` capability declarations.
